# Code-Order - 开发顺序技能

## 🎯 触发条件

**每次执行任何开发或编辑任务时都必须遵循**

无论是编写新功能、修复bug、重构代码，还是编辑配置文件，都必须按此顺序执行。

## 📋 核心功能

本技能规范了开发任务的完整流程，确保开发工作有序、高效、可追溯。

## 🔄 开发流程步骤

### 步骤1：读取项目协作文件

**操作**：读取 `agent_project.md`

**目的**：
- 了解项目整体情况
- 查看当前任务进度
- 了解项目技术栈和规范
- 查看是否有其他Agent正在处理相关任务

**输出**：对项目现状的清晰认知

---

### 步骤2：了解项目情况

**操作**：
- 浏览项目结构
- 阅读相关代码文件
- 了解依赖关系
- 查看文档（README、ops.md等）

**目的**：
- 理解项目架构
- 熟悉代码风格
- 掌握现有功能
- 避免重复开发

**输出**：对项目技术细节的掌握

---

### 步骤3：领取或创建任务

**操作**：
- 在 `agent_project.md` 中找到合适的任务
- 或创建新任务
- 在任务看板中标记执行人
- 在交流区留言说明

**目的**：
- 明确工作目标
- 避免任务冲突
- 记录工作分配
- 便于进度追踪

**示例留言**：
```
[2026-03-02 18:00] 琉璃: 领取T007任务 - MCP服务扩展
计划新增用户认证、文章管理等模块的MCP工具接口
```

---

### 步骤4：进行项目开发

**操作**：
- 编写代码
- 遵循项目开发规范
- 添加必要的注释
- 遵循其他开发守则（如ops.md中的规范）

**注意事项**：
- 保持代码风格一致
- 编写清晰的注释
- 考虑错误处理
- 遵循安全最佳实践

---

### 步骤5：更新开发进度和文档

**操作**：
- 在 `agent_project.md` 中更新任务进度
- 记录已完成的工作
- 记录遇到的问题和解决方案
- 更新变更日志
- **更新相关开发文档**（重要！）

**文档更新要求**：

根据修改内容，更新 `docs/` 目录下的相应文档：

| 修改类型 | 需要更新的文档 | 更新内容 |
|----------|----------------|----------|
| 新增/修改API接口 | `docs/api.md` | 接口路径、方法、参数、响应 |
| 架构变更 | `docs/architecture.md` | 架构图、组件说明、数据模型 |
| 数据库变更 | `docs/database.md` | 表结构、索引、关系 |
| 部署配置变更 | `ops.md` | 部署步骤、环境变量 |
| MCP服务变更 | `docs/mcp.md` | 工具列表、使用示例 |
| 代码规范变更 | `docs/development.md` | 规范说明、示例代码 |

**文档更新示例**：
```markdown
## 新增接口记录

### POST /api/posts/{id}/publish
- **功能**: 发布文章
- **鉴权**: 需要登录（作者或超管）
- **请求**: 无
- **响应**: 更新后的文章对象
- **错误码**: 401(未认证), 403(无权限), 404(文章不存在)
```

**目的**：
- 让其他Agent了解进度
- 记录工作历史
- 便于问题追溯
- 维护项目透明度
- **保持文档与代码同步**

**示例更新**：
```markdown
| 任务ID | 任务名称 | 进度 | 执行人 | 备注 |
|--------|----------|------|--------|------|
| T007 | MCP服务扩展 | 50% | 琉璃 | 用户认证模块完成，已更新docs/api.md和docs/mcp.md |
```

---

### 步骤6：交流区沟通（如有需要）

**操作**：
- 在交流区留言
- 汇报重要进展
- 提出阻塞问题
- 协调其他Agent

**场景**：
- 遇到技术难题需要讨论
- 需要其他Agent协助
- 重要决策需要同步
- 任务完成汇报

---

### 步骤7：重复直到完成

**操作**：
- 重复步骤4-6
- 逐步完成任务
- 持续更新进度
- 直到任务100%完成

## 🎯 关键原则

1. **先了解，后开发** - 不盲目动手
2. **先记录，后执行** - 明确任务目标
3. **边开发，边更新** - 保持进度透明
4. **遇问题，多交流** - 及时沟通协调
5. **代码文档同步** - 修改代码必更新文档

## 📝 文档编写指南

### 何时需要更新文档

**必须更新文档的场景**：
- ✅ 新增API接口
- ✅ 修改现有接口（路径、参数、响应变更）
- ✅ 数据库表结构变更
- ✅ 架构组件变更
- ✅ 新增MCP工具
- ✅ 部署配置变更
- ✅ 代码规范变更

**可选更新文档的场景**：
- ⚪ 修复bug（不影响接口行为）
- ⚪ 代码重构（行为不变）
- ⚪ 样式调整

### 文档编写规范

**通用规范**：
1. 使用Markdown格式
2. 文档顶部注明文档位置、作用、维护者
3. 使用表格展示结构化信息
4. 代码块标明语言类型
5. 重要内容使用引用块（`>`）突出

**API文档规范**（`docs/api.md`）：
```markdown
### 接口名称

| 属性 | 说明 |
|------|------|
| 路径 | `/api/xxx/xxx` |
| 方法 | GET/POST/PUT/DELETE |
| 鉴权 | 是/否 |

**请求参数**：
| 参数 | 类型 | 必需 | 说明 |
|------|------|------|------|
| xxx | string | 是 | 说明 |

**响应示例**：
```json
{
  "success": true,
  "data": {}
}
```
```

**架构文档规范**（`docs/architecture.md`）：
- 使用ASCII图或文字描述架构
- 每个组件说明职责和核心模块
- 数据模型使用JSON格式示例
- 核心流程使用流程图描述

### 文档更新检查清单

开发任务完成后，检查以下文档是否需要更新：

- [ ] `docs/api.md` - 是否有新增/修改的API？
- [ ] `docs/architecture.md` - 架构是否有变更？
- [ ] `docs/database.md` - 数据库是否有变更？
- [ ] `ops.md` - 部署配置是否有变更？
- [ ] `docs/mcp.md` - MCP服务是否有变更？
- [ ] `docs/development.md` - 开发规范是否有变更？
- [ ] `README.md` - 项目介绍是否需要更新？
- [ ] `agent_project.md` - 任务进度是否已更新？

## 💡 最佳实践

- 小步快跑：每次更新都记录进度，哪怕只完成了10%
- 及时沟通：遇到问题不要闷头苦干，及时在交流区求助
- 保持整洁：代码和文档都要保持优雅整洁
- 尊重规范：遵循项目已有的开发规范和风格
