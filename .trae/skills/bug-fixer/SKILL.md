# Bug修复工程师标准化流程

> **Skill名称**: bug-fixer
> **适用角色**: 开发工程师
> **使用场景**: 修复Bug任务时

---

## 修复流程总览

```
┌─────────────────────────────────────────────────────────────────┐
│  阶段1: 查询阶段                                                  │
│  ├── 从agent_project.md查询分配给自己的未解决Bug                   │
│  ├── 阅读Bug详情和关联的测试用例                                   │
│  └── 输出：Bug任务清单                                             │
├─────────────────────────────────────────────────────────────────┤
│  阶段2: 分析阶段                                                  │
│  ├── 阅读相关代码（出问题的模块）                                  │
│  ├── 阅读测试用例（理解预期行为）                                  │
│  ├── 复现Bug（确认问题）                                          │
│  └── 输出：问题分析笔记                                            │
├─────────────────────────────────────────────────────────────────┤
│  阶段3: 修复阶段                                                  │
│  ├── 编写修复代码                                                  │
│  ├── 自我Code Review                                              │
│  ├── 本地验证修复                                                  │
│  └── 输出：修复后的代码                                            │
├─────────────────────────────────────────────────────────────────┤
│  阶段4: 测试阶段                                                  │
│  ├── 执行关联的测试用例                                            │
│  ├── 执行相关回归测试                                              │
│  ├── 确认所有测试通过                                              │
│  └── 输出：测试执行记录                                            │
├─────────────────────────────────────────────────────────────────┤
│  阶段5: 更新阶段                                                  │
│  ├── 更新agent_project.md标记Bug为已解决                           │
│  ├── 在交流区汇报修复结果                                          │
│  ├── 如无法修复则标记为搁置并说明原因                               │
│  └── 输出：Bug状态更新                                             │
└─────────────────────────────────────────────────────────────────┘
```

---

## 详细流程步骤

### 阶段1: 查询阶段 - 获取Bug任务

#### 1.1 查询未解决的Bug

**在 agent_project.md 中查找**：

1. 找到 `### 🐛 【Bug修复】任务` 部分
2. 筛选出状态为 `🔴 待修复` 且负责人为自己的Bug
3. 按严重程度（P0 > P1 > P2）排序

#### 1.2 阅读Bug详情

**必须阅读的信息**：
- [ ] Bug-ID 和 所属模块
- [ ] 严重程度（P0/P1/P2/P3）
- [ ] 问题描述
- [ ] 复现步骤
- [ ] 预期结果 vs 实际结果
- [ ] **关联测试用例位置**（测试用例文档行号）
- [ ] **测试脚本位置**（如果有自动化测试）
- [ ] 测试报告链接

#### 1.3 输出Bug任务清单

```markdown
## Bug修复任务清单

**当前时间**: YYYY-MM-DD HH:MM
**负责人**: [你的名字]

### 待修复Bug列表

| 序号 | Bug-ID | 模块 | 严重程度 | 问题简述 | 测试用例位置 | 优先级 |
|------|--------|------|----------|----------|--------------|--------|
| 1 | BUG-001 | auth | P0 | 注册接口500错误 | auth-api-testcases.md#L15-L25 | 高 |
| 2 | BUG-002 | auth | P0 | 登录接口500错误 | auth-api-testcases.md#L45-L55 | 高 |

### 修复计划
1. 先修复P0级别的Bug
2. 按模块集中修复（减少上下文切换）
3. 每个Bug修复后立刻执行关联测试用例
```

---

### 阶段2: 分析阶段 - 理解问题

#### 2.1 阅读相关代码

**根据Bug所属模块阅读**：
- [ ] 路由代码：`backend/app/routers/{module}.py`
- [ ] 模型代码：`backend/app/models/{module}.py`
- [ ] 服务代码：`backend/app/services/`（如有）
- [ ] 数据库操作代码

#### 2.2 阅读测试用例

**根据Bug中记录的测试用例位置**：

```markdown
## 测试用例分析

**Bug-ID**: BUG-XXX
**测试用例位置**: [文件路径#L行号](file:///path/to/file.md#L行号)

### 测试用例内容
| 用例ID | 场景 | 操作步骤 | 预期结果 |
|--------|------|----------|----------|
| TC-001 | ... | ... | ... |

### 关键验证点
1. 
2. 
3. 

### 问题分析
- 预期行为：
- 实际行为：
- 可能原因：
```

**手动复现步骤**：
1. 启动后端服务

3. 确认能稳定复现问题
4. 记录错误日志和堆栈信息

```markdown

**Bug-ID**: BUG-XXX
**复现时间**: 
**复现结果**: 可复现 / 不可复现

```

### 阶段3: 修复阶段 - 编写修复代码

#### 3.1 编写修复

**修复原则**：
- 最小化修改，只修复问题本身
- 保持代码风格一致
- 添加必要的注释说明修复原因
- 考虑边界情况

**修复检查清单**：
- [ ] 修复了根本原因，不只是症状
- [ ] 没有引入新的问题
- [ ] 代码风格与项目一致
- [ ] 添加了必要的错误处理
- [ ] 考虑了并发情况（如适用）

#### 3.2 自我Code Review

**Review检查清单**：
- [ ] 修复逻辑是否正确？
- [ ] 是否有更优雅的解决方案？
- [ ] 是否处理了所有边界情况？
- [ ] 是否会影响其他功能？
- [ ] 性能是否有影响？

```markdown
## 修复代码Review记录

**Bug-ID**: BUG-XXX
**Reviewer**: 自己
**Review时间**: 

### 修改内容
```python
# 修改前
[原代码]

# 修改后
[新代码]
```

### Review检查
- [ ] 修复逻辑正确
- [ ] 边界情况处理
- [ ] 无性能影响
- [ ] 无副作用

### Review结论
- [ ] 通过，可以提交测试
- [ ] 需要调整（说明：...）
```

### 阶段4: 测试阶段 - 验证修复

#### 4.1 执行关联测试用例

**根据Bug中记录的测试用例执行**：

```bash
# 如果是Python测试脚本
python docs/tests/backend/test_xxx.py

# 或者手动执行curl测试
curl -X POST http://localhost:8002/api/xxx \
  -H "Content-Type: application/json" \
  -d '{...}'
```

#### 4.2 执行回归测试

**测试相关功能是否受影响**：
- [ ] 同模块的其他功能
- [ ] 依赖此功能的上游功能
- [ ] 此功能依赖的下游功能

#### 4.3 记录测试结果

```markdown
## Bug修复测试记录

**Bug-ID**: BUG-XXX
**修复时间**: 
**测试时间**: 

### 关联测试用例执行结果

| 用例ID | 场景 | 结果 | 备注 |
|--------|------|------|------|
| TC-001 | ... | ✅ 通过 | |
| TC-002 | ... | ✅ 通过 | |

### 回归测试结果

| 测试项 | 结果 | 备注 |
|--------|------|------|
| 同模块功能A | ✅ 通过 | |
| 同模块功能B | ✅ 通过 | |

### 测试结论
- [ ] 所有测试通过，修复成功
- [ ] 有测试未通过，需要继续修复
```

### 阶段5: 更新阶段 - 更新Bug状态

#### 5.1 测试通过 - 标记为已解决

**在 agent_project.md 中更新**：

1. 找到对应的Bug任务行
2. 更新进度为 `100%`
3. 更新状态为 `✅ 已解决`
4. 在备注中添加修复说明

```markdown
| BUG-001 | 用户注册接口500错误 | 100% | @糖衣 | ✅ 已解决 | [测试报告](path/to/report)，修复：数据库连接问题 |
```

**在交流区汇报**：

```markdown
```

**Bug-ID**: BUG-XXX
**问题**: [简述]
**修复内容**: 
- 
- 

**测试结果**: 
- 关联用例：X个全部通过
- 回归测试：通过

**验证方式**:
1. 
2. 

(修复完成，请测试工程师验证)
```
```

#### 5.2 测试不通过 - 继续修复

**如果测试未通过**：
1. 回到阶段3继续修复
2. 记录新的发现和调整
3. 重新执行测试

#### 5.3 无法修复 - 标记为搁置

**如果确实无法修复**：

1. 更新Bug状态为 `⏸️ 已搁置`
2. 在备注中说明原因
3. 在交流区说明情况

```markdown
| BUG-001 | 用户注册接口500错误 | 0% | @糖衣 | ⏸️ 已搁置 | 原因：需要重构底层架构，建议下个版本处理 |

## 重要提示

### 关于测试用例位置

**测试工程师会在Bug中记录**：
- 测试用例文档位置（精确到行号）
- 测试脚本位置（如果有自动化测试）
- 用例ID（如 REG-001）

**你必须**：
1. 根据这些位置找到对应的测试用例
2. 理解测试的预期行为
3. 修复后执行这些测试用例验证

### 关于Bug严重程度

| 严重程度 | 处理优先级 | 响应时间 |
|----------|------------|----------|
| P0 | 最高 | 立即处理 |
| P1 | 高 | 24小时内 |
| P2 | 中 | 本周内 |
| P3 | 低 | 下个迭代 |

### 关于无法修复的情况

**可以标记为搁置的情况**：
- 需要架构级重构才能修复
- 依赖外部服务/库的问题
- 修复成本远高于收益
- 风险过高，可能引入更多问题

**必须说明**：
- 尝试过的修复方法
- 无法修复的具体原因
- 对项目的影响评估
- 建议的替代方案或处理时间
